# requirements.md — タスク機構（#128 / 再定義: セッション内ゴール到達管理）

親 Issue: [#128](https://github.com/sugimomoto/CoworkAgentForKintone/issues/128)（size L / tier 2-foundation / P3）
関連: #83（Agentic iPaaS の"確認"面）, 既存の Tool Confirmation（HITL）, アーティファクト機構, 会話/セッション永続。

## 0. 重要な再定義（2026-07-11）

本 Issue は当初 **Cowork の Scheduled/Recurring Tasks（常設スケジュール台帳）** として記述されていたが、本来意図していたのは **「1つのゴールに向かう 1 セッションが、完了に到達するまでの作業を分解・進捗管理・承認する仕組み」**（＝**概念B**）である。

- **対象（概念B）**: plan（サブタスク分解）→ 承認 → 逐次実行 → 進捗可視化 → 完了、を**セッション内**で管理する。
- **対象外（概念A）**: cron 常設・Active/Paused・実行履歴の永続台帳（それは Deployments #81 の領分）、Webhook 起動（#114）、**永続タスク台帳（Memory Store / kintone アプリ）**。
- 帰結: **永続ストアは不要**。状態は**セッションスコープ**で、既存の会話/セッション/アーティファクト永続に乗る。**#15（Memory Store）への依存は無い**（独立トラック）。

## 1. 概要

複雑な依頼に対して、エージェントが「どう進めるか（サブタスクの計画）」を**明示的に宣言**し、ユーザーが**確認・承認**してから実行、実行中は**進捗（どのステップか / 完了・残り）**を可視化し、破壊的操作の手前では**承認ゲート**を挟み、最後に**成果サマリ**を返す。長い agent 実行を **legible（見える）かつ controllable（止められる）**にすることが狙い。#83 の「AI がフローを生成・ユーザーが確認・実行」の"確認"面に対応する。

## 2. スコープ

### 2.1 In Scope

1. **Plan（サブタスク分解）の宣言・更新**
   - エージェントがゴールをサブタスクに分解し、状態付きで宣言/更新する仕組み（Custom Tool `update_plan` 的、TodoWrite 相当）。
2. **Plan / 進捗の UI**
   - チェックリスト状の進捗パネル（各サブタスク: `pending / running / done / skipped`）。現在ステップと残りが一目で分かる。
3. **HITL（承認）**
   - **plan の事前承認/修正**（実行前に「この計画で進める」を確認）。
   - **破壊的操作の承認ゲート**は既存の Tool Confirmation（`always_ask` / `DESTRUCTIVE_TOOL_NAMES`）を流用。
4. **完了サマリ + 成果物リンク**
   - 完了時に「やったこと・生成物」を要約表示。
5. **状態の永続はセッションスコープ**
   - plan/進捗は会話の一部として既存永続（会話/セッション/アーティファクト）に乗せ、**リロード/再開で復元**する。新規ストアは作らない。

### 2.2 Out of Scope

- cron 常設スケジュール（→ Deployments #81）、Webhook 起動（→ #114）
- 永続タスク台帳（Memory Store / kintone アプリ）、Active/Paused、跨セッション実行履歴
- Memory Store 統合（#15・独立トラック）

## 3. ユーザーストーリー

- **US-1**: 複雑な依頼をすると、エージェントが「まずこう進めます」と**サブタスクの計画を提示**し、承認してから実行してくれる。
- **US-2**: 実行中、**今どのステップで、何が完了し、何が残っているか**が見える。
- **US-3**: 破壊的操作の直前で**承認**を求められ、止められる。
- **US-4**: 完了時に**やったこと・成果物の要約**が出る。会話を開き直しても進行状況が残っている。

## 4. 受け入れ条件

- [ ] エージェントが plan を宣言でき、UI にサブタスクが**状態付き**で表示される。
- [ ] ユーザーが plan を**承認/修正**してから実行に進める。
- [ ] 実行に伴いサブタスクの状態が `pending → running → done`（or `skipped`）に更新され、UI に反映される。
- [ ] 破壊的操作の手前で承認カードが出る（既存 confirmation 流用）。
- [ ] 会話をリロード/再開しても、進行中の plan/進捗が復元される。
- [ ] 既存機能（会話/ツール実行/アーティファクト/承認）にデグレなし。型/lint/test/build グリーン。

## 5. 制約 / 既存資産の活用

- **状態はセッションスコープ**。新規の永続ストア（Memory/kintone アプリ）は不要。**既存の会話・セッション・アーティファクト永続**に乗せる。
- **HITL は既存 Tool Confirmation を土台**に、「plan 承認 / （任意で）step 承認」へ拡張。
- **plan の表現は既存アーティファクト機構**（`agent-draft` / `kintone-customize-bundle` と同型の "plan" kind）を第一候補に検討。
- 本機能は**新規の外部通信を増やさない**（secret-zero 等の追加考慮は不要）。

## 6. 決定事項（2026-07-11 確定）

- **A. plan 生成方式 = (a) Custom Tool `update_plan`**。エージェントが TodoWrite 的にサブタスクを宣言・更新する。plan アーティファクト提案（propose_plan）は採らない。
- **B. 承認の粒度 = 破壊的操作のみ**。plan 全体承認・ステップ承認は設けず、**既存 Tool Confirmation（`always_ask` / `DESTRUCTIVE_TOOL_NAMES`）のまま**。＝ `update_plan` 自体は非破壊・自動実行（承認カードを出さない）。
- **C. 対象 = 全エージェント**（全 built-in + custom に `update_plan` を公開）。
- **D. plan 生成は任意**。毎回強制せず、ユーザー依頼の複雑度に応じてエージェントが判断して立てる（軽い依頼では plan を出さない）。システムプロンプトで「多段・複数ファイル・破壊的操作を含む依頼のときに使う」と誘導する。
