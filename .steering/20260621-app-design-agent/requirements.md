# 要求定義: アプリ設計・構築支援エージェント (built-in 4th variant) — #117

## 概要
業務内容や既存資料 (PDF / Excel / 紙フォーム) から **kintone アプリを設計提案し、構築まで伴走**する
標準 (built-in) エージェントを追加する。エージェントデザイナー(#48) が「**エージェント**」を設計するのに対し、
本エージェントは「**アプリ (テーブル設計・ビュー・ワークフロー・権限) そのもの**」を設計・構築する。

- 対象 Issue: [#117](https://github.com/sugimomoto/CoworkAgentForKintone/issues/117)
- 利用: [#24](../../issues/24) 管理系ツール (create-app / fields / layout / views / process / acl / plugins / deploy)
- 類似パターン: [#48](../../issues/48) エージェントデザイナー (`propose_agent` → 提案 → 永続化)

## 背景・狙い
- 「作る側」(情シス/管理者) の中核作業 = アプリ設計。要件や既存帳票を kintone の構造に落とすのは手間が大きい。
- #24 で管理系ツールを追加したので、それを使ってアプリを**実際に組み立てられる**エージェントを標準提供し、
  「相談 → 設計 → 構築 → 反映」を 1 つの会話で完結させる。

## ペルソナ
- 情シス / kintone 管理者 (アプリ管理者権限を持つ)。業務の要件はわかるが、フィールド設計やワークフロー設定の
  手数を減らしたい。

## ユーザーストーリー
- **US-1**: admin が「この帳票(PDF)を kintone アプリにしたい」と添付 → エージェントが項目を読み取り、
  フィールド・レイアウト・必要なら一覧/ワークフローの**設計案を提示**する。
- **US-2**: 設計案をレビュー・修正依頼 → 合意後、エージェントが **preview にアプリを構築**し、差分を説明。
- **US-3**: 「本番に反映して」で deploy、デプロイ状況も確認できる。
- **US-4**: 既存アプリに対して「フィールドを追加」「ワークフローを足す」といった**増築**も依頼できる。
- **US-5**: 一般ユーザにはこのエージェントが**見えない / 使えない** (admin 専用)。

## 受け入れ条件
- [ ] AC-1: built-in の 4 つ目の variant として追加され、Header / 一覧に出る (admin のみ)。
- [ ] AC-2: 添付資料 (PDF / Excel 等) を読み取り、アプリ設計案を提示できる。
- [ ] AC-3: 合意後に #24 ツールで preview にアプリを構築し、deploy で反映できる。
- [ ] AC-4: 既存アプリの増築 (フィールド/ビュー/プロセス追加) ができる。
- [ ] AC-5: **admin 専用**: 非 admin には公開されない (管理系ツールの gating と整合)。
- [ ] AC-6: 破壊的操作 (deploy / フィールド削除) は承認カードが出る (#24 既存)。
- [ ] AC-7: 既存 3 variant (業務 / デザイナー / カスタマイザー) と役割が衝突しない (system prompt で明示)。
- [ ] AC-8: Worker / Plugin unit test green。

## 制約・前提
- **管理系ツールが必要**: 本エージェントは #24 の管理系ツール群を使う。ただし #24 ではこれらを
  「admin 専用・Custom Agent でのみ」に gating した。**built-in でこれらをどう許可し、admin に限定するか**が
  最大の設計論点 (→ design.md で確定。built-in には現状 ACL が無い)。
- **OAuth スコープ**: 追加不要 (`k:app_settings:write` / `k:file:write` 付与済み)。
- **kintone 権限**: アプリ管理者権限が要る。無ければ API が 403 を返す (kintone 側で担保)。
- **安全性**: 直接 deploy せず preview → 人間レビュー → deploy を基本フローにする。

## 主要な設計判断 (確定 2026-06-21)
1. **admin ゲートは不要**: kintone 側でアプリのカスタマイズはアプリ管理者権限が必須 (非 admin は API が 403)。
   よって**エージェント側で admin 振り分けはしない**。本 variant は管理系ツールを含む通常の public built-in として追加し、
   安全性は kintone の権限に委ねる。(= AC-5「admin 専用」は kintone 権限による実効的な制限に読み替え。
   エージェント自体は全ユーザに見えてよい。)
2. **設計提案は「会話 + ツール直実行」**: `propose_app` のような構造化提案 UI は作らない。会話で設計を説明し、
   合意後に #24 ツールを直接呼んで preview 構築 → deploy。安全は preview→レビュー→deploy のフローと
   既存の承認カード (deploy / フィールド削除) で担保。
3. **資料読解は Anthropic skills 任せ**: `pdf` / `xlsx` / `docx` skill を attach し、添付資料の読解はそれに委ねる。
4. **モデル: Opus**。

## 非対象 (Out of Scope)
- 一般業務ユーザへの公開。
- アプリ設計の自動最適化や大規模な提案レビュー専用 UI (MVP では会話 + 既存 artifact で start、段階拡張)。
- #24 管理系ツール自体の追加実装 (済み)。
