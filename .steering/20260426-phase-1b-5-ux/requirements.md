# Phase 1b-5 — UX 強化 要求定義

## 背景

Phase 1b-4 までで kintone 読み取り 4 + 書き込み 6 = 計 **10 ツール** が Agent から呼べるようになり、機能的には MVP の F-01〜F-06 をカバーしている。
一方、PRD の受入条件のうち以下が **UI として未対応 / 不足**:

- **F-07 (HITL)**: 破壊的操作の計画提示と承認 → 現状はプロンプトの guardrail に依存し、UI ボタンが無い
- **F-08 (進捗表示)**: tool 実行中の状態が「考え中…」のドット表示のみ。どのツールを何の引数で呼んでいるか見えない
- **エラー回復**: API Key 未設定 / OAuth 失効 / Worker 5xx 時の **回復導線** (再連携・設定画面リンク・再試行) が薄い
- **送信中のキャンセル / 新規セッション開始** などの基本操作の欠落

本フェーズで上記の UX ギャップを埋めて MVP リリース水準に揃える。
ロジック的な機能追加 (新ツール / 新エンドポイント) は **行わない** — UI 層と event interpreter の拡張に閉じる。

## スコープ全体図

| カテゴリ | 概要 |
|---|---|
| 1. ツール実行可視化 | tool_use を「ツール実行カード」として表示し、進捗・成功・失敗を 1 つの単位で見せる |
| 2. HITL 承認 UI (F-07) | 破壊的ツールを `always_ask` にし、確認ボタンを UI に出す |
| 3. エラー UX / 回復ガイド | 認証 / 通信エラーをチャット内で案内 + 回復ボタン |
| 4. 入力体験 | 送信中の disable + キャンセル |
| 5. セッション操作 | 新規セッション開始ボタン |

優先度: **P1 = 本フェーズ必須**、**P2 = 余力で着手**、**P3 = 別フェーズ送り**。
工数: S = 半日 / M = 1〜2 日 / L = 数日。

---

## 機能要件

### F1. ツール実行の可視化

| ID | 内容 | P | 工数 |
|---|---|---|---|
| F1-1 | `agent.custom_tool_use` / `mcp_tool_use` を **ツール実行カード** として表示 (ツール名 + 引数のサマリ) | P1 | M |
| F1-2 | tool_use → tool_result のリンク (成功/失敗/件数) を **同一カード内** に集約表示 | P1 | S |
| F1-3 | 失敗 tool_call は **赤系** で表示 + エラー本文を折り畳み | P1 | S |
| F1-4 | 長時間 fetch 中の **スピナー / 経過秒数** (現状 `agent.thinking` ドットのみ) | P2 | S |

**受入条件**
- レコード追加・更新・削除を依頼すると、**何のツールが何の引数で呼ばれたか** がカードで見える
- ツールが失敗した場合、エラー文がチャット履歴に残り、後から確認できる

### F2. HITL 承認 UI (F-07)

| ID | 内容 | P | 工数 |
|---|---|---|---|
| F2-1 | 破壊的ツール (`update-record` / `update-records` / `delete-records`) を **`permission_policy: always_ask`** に切替 | P1 | S |
| F2-2 | `tool_confirmation_required` イベントを受けて **承認 / 却下ボタン** を出す | P1 | M |
| F2-3 | 承認時に引数 (例: 削除対象 ID 一覧) を確認できる詳細表示 | P2 | M |
| F2-4 | 「以降このツールは自動承認」のセッション内オプション | P3 | S |

**受入条件**
- レコード更新・削除前に **計画 (操作内容と対象) が提示** され、承認ボタンを押すまで実行されない
- 却下ボタンを押すと処理が中止され、Agent に「拒否された」ことが伝わる

### F3. エラー UX / 回復ガイド

| ID | 内容 | P | 工数 |
|---|---|---|---|
| F3-1 | API Key 未設定 / 無効 → **設定画面への CTA リンク** をチャット内に表示 | P1 | S |
| F3-2 | OAuth 失効 → **「再連携」ボタン** を Composer 上部にバナー表示 | P1 | M |
| F3-3 | Worker 5xx / network エラー → **"Retry" ボタン** | P2 | S |
| F3-4 | session 終了 (terminated) を可視化 + **「新しいセッションを開始」** ボタン | P2 | S |

**受入条件**
- API Key 未設定状態でチャットを開いても、ユーザがどこを直せばよいか UI から自明
- OAuth 失効時、設定画面に行かなくてもチャット側から再連携を起動できる

### F4. 入力体験 (Composer)

| ID | 内容 | P | 工数 |
|---|---|---|---|
| F4-1 | 送信中の **入力欄 disable + キャンセルボタン** (実行中の Agent ターンを中断) | P1 | S |
| F4-2 | Shift+Enter で改行、Enter で送信 (現状の挙動を確認のうえ未対応なら実装) | P2 | S |
| F4-3 | 文字数 / トークン目安カウンタ | P3 | S |

**受入条件**
- ユーザは Agent 応答を待っている間に **キャンセル** できる (= 次のメッセージを送れる)

### F5. セッション操作

| ID | 内容 | P | 工数 |
|---|---|---|---|
| F5-1 | **「新しいセッションを開始」** ボタン (Header 内) | P1 | S |
| F5-2 | HistoryView で **過去セッション一覧 → 切替** | P2 | M |
| F5-3 | パネル開閉状態の永続化 (localStorage) | P3 | S |

**受入条件**
- ユーザは現在のセッションを終了して **新しい会話を始められる**

## 非対象 (本フェーズではやらない)

- **Markdown レンダリング** (見出し / コードブロック / リスト) — Phase 1c 候補
- **レコード一覧の table 表示** — Phase 1c 候補 (LLM 出力の構造化が前提)
- **Suggested prompts** — Phase 2 候補
- **モバイル対応** — Phase 1c 以降
- 新しい kintone ツールの追加
- アーキテクチャ変更 (Worker / Agent 構成は維持)

---

## 受け入れ条件 (フェーズ全体)

P1 項目すべての受入条件を満たすこと。具体的には:

- [ ] レコード追加・更新・削除のシナリオで、**呼ばれたツールと引数** がチャット履歴上に残る (F1-1, F1-2)
- [ ] ツール失敗時にエラー内容が画面で確認できる (F1-3)
- [ ] **更新・削除を依頼すると承認ボタンが出る** (F2-1, F2-2)
- [ ] 却下を選ぶと Agent が中止する (F2-2)
- [ ] API Key 未設定で開いた際、**設定画面へのリンクまたは案内** が UI に出る (F3-1)
- [ ] OAuth 失効時、**チャット画面から再連携** を開始できる (F3-2)
- [ ] Agent 応答中に **キャンセル** できる (F4-1)
- [ ] **新規セッション開始** ボタンが Header にある (F5-1)

---

## 制約 / 前提

- 本フェーズは **UI 層 + event interpreter の拡張** に閉じる。Worker / kintone-mcp / OAuth 経路には触らない
- `agent_toolset_20260401` および `mcp_toolset` の `permission_policy` 仕様 (`always_allow` / `always_ask`) を活用する。Anthropic API の仕様変更があった場合は影響範囲を再評価
- 新しい依存ライブラリ (Markdown renderer 等) は本フェーズでは入れない
- 既存テスト (267 plugin / 80 mcp) を割らないこと
